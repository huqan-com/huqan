## Gate Mapping

### AB1 — Action Risk Classifier

**Where it should apply:** Every `tools/call` request, before `callTool()` execution.

**Current state:** Not wired into MCP runtime.

**What AB1 would classify:**
- `axiom.learn` → elevated (graph mutation)
- `axiom.ask` → safe (read-only)
- `axiom.verify` → safe (read-only)
- `axiom.plan` → safe (read-only)
- `axiom.agent` → elevated (runs agent loop, may invoke tools)
- `axiom.policy` → safe (read-only)
- `axiom.approvals` → safe (read-only)
- `axiom.reason` → safe (read-only)
- `axiom.compare` → safe (read-only)
- `axiom.dream` → safe (read-only, speculative)

### AB2 — Tool Call Gate

**Where it should apply:** Every `tools/call` request, after AB1 classification.

**Current state:** Not wired into MCP runtime.

**What AB2 would gate:**
- All tool calls go through AB2 before dispatch
- `axiom.learn` → requires approval (graph mutation)
- `axiom.agent` → requires review (may invoke tools)
- All read-only tools → auto-allow

### AB3 — Code Change Gate

**Where it should apply:** Not directly applicable to MCP runtime. AB3 gates code changes, not MCP tool calls.

**Current state:** Not applicable for MCP private alpha.

**Note:** If MCP tools ever modify code files, AB3 would apply. Currently none do.

### AB4 — Memory Mutation Gate

**Where it should apply:** `axiom.learn` only.

**Current state:** Not wired into MCP runtime.

**What AB4 would gate:**
- `axiom.learn` → approve/block based on schema validity, tombstone integrity
- All other tools → not applicable (read-only)

### AB5 — Automation Safety Gate

**Where it should apply:** Any MCP operation that could trigger automation (merge, deploy, tag, release, auto-push).

**Current state:** No MCP tools do this. AB5 is a safety net.

**What AB5 would gate:**
- Currently nothing in MCP triggers automation
- AB5 blocks if someone tries to add auto-merge/deploy/tag/release tools
- AB5 is the firewall against future dangerous tool additions

### AB6 — Sandbox Isolation Gate

**Where it should apply:** Any MCP operation that executes untrusted code.

**Current state:** No MCP tools execute code directly. `axiom.agent` runs in-process.

**What AB6 would gate:**
- If `axiom.agent` is extended to run sandboxed code, AB6 applies
- If MCP tools are added that execute user-provided code, AB6 applies
- AB6 is the firewall against future code execution

## Safety Invariants

These must hold after V2.6 integration:

1. **Unknown MCP tool cannot silently execute.** Unknown tools default to block.

2. **Mutating MCP tool cannot bypass AB2.** `axiom.learn` must pass AB2 before execution.

3. **Code-changing MCP operation must pass AB3.** Currently none exist. AB3 is the firewall.

4. **Memory/canonical mutation must pass AB4.** `axiom.learn` must pass AB4 before graph write.

5. **Merge/deploy/tag/release/autopush must pass AB5 and default block.** Currently none exist in MCP. AB5 is the firewall.

6. **Sandbox/rollback-sensitive operation must pass AB6.** Currently none exist in MCP. AB6 is the firewall.

7. **Private alpha must default to read-only or dry-run-only for risky actions.** Only `axiom.ask`, `axiom.verify`, `axiom.plan`, `axiom.policy`, `axiom.approvals`, `axiom.reason`, `axiom.compare`, `axiom.dream` are auto-allow.

8. **No token/secret should be logged.** Gate decisions must not expose credentials.

9. **No gate output should cause automatic execution by itself.** Gates classify and advise. Execution requires explicit downstream action.

10. **Human approval remains required for irreversible operations.** `axiom.learn` (graph mutation) requires human approval in private alpha.

**Golden rule:** MCP may expose capability, but HUQAN decides whether the action crosses the boundary.

## Open Questions

1. **Should `axiom.agent` be allowed or dry-run-only in private alpha?** It runs an agent loop that may invoke internal tools. Dry-run is safer.

2. **Should `axiom.learn` require per-call approval or batch approval?** Per-call is safer but more friction. Batch is convenient but riskier.

3. **Should we add an `axiom.gate` read-only tool?** Would allow external tools to inspect gate decisions without executing. Useful for demo.

4. **Should the gate adapter be a separate module or inline in `mcpServer.js`?** Separate module is cleaner for testing. Inline is simpler.

5. **Should private alpha log gate decisions?** Useful for debugging but must not log secrets.

## Final Recommendation

V2.6 should be incremental:

1. **PR0 (now):** Inventory + plan. Docs only.
2. **PR1:** Pure gate adapter. No MCP changes.
3. **PR2:** Wire adapter into MCP. Dry-run enforcement.
4. **PR3:** Demo + docs for private alpha.

This order ensures:
- Each PR is small and reviewable
- No MCP runtime changes until gate adapter is tested
- Dry-run mode protects against mistakes in alpha
- Demo comes last, after everything works

**V2.6 goal:** MCP private alpha where HUQAN stops risky actions before they execute.
